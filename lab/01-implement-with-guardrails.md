# Lab 01: 機能実装でガードレールを体験する

**テーマ:** 依存追加を伴う機能実装で、ルール・手順・検証が連携する様子を見る

## シナリオ

Outdoor eShop に商品詳細ページへの **画面遷移(ルーティング)** を追加する。
React SPA でルーティングを実現するには新しいランタイム依存 `react-router-dom` が必要になる。
この「依存を追加する」場面で、リポジトリに仕込まれた役割分担が働く様子を確認する。

| 層 | 実体 | 役割 |
| --- | --- | --- |
| プロジェクトの前提 | `.github/copilot-instructions.md` | 実行環境・技術スタック・依存ルールを提示 |
| SDLC ルール | `AGENTS.md` | GitHub Flow、Human-in-the-loop、Issue 対応、Skill 起動条件を提示 |
| 実行手順 | `.github/skills/` | 依存追加と Git ワークフローの具体的な手順を提示 |
| 即時検証 | `.github/hooks/` | 依存・環境ファイル編集直後に自動チェック、特定のファイルアクセス制御 |
| Pull Request 時の検証 | `.github/workflows/ci.yml` | Lab 02 で `npm ci`、test、build を再確認 |

## 前提条件

- Lab 00 を完了していること。
- 開発サーバーが起動し、Browser Canvas で表示できること。

## 手順
※GPT-5.6 Sol, High, Default agent の設定が推奨。

### 1. 機能追加を依頼する
Agent の検証動作に衝突しないよう、一度 [Stop] ボタンから現在起動中の開発サーバーを停止する。
Plan モードに切り替えて、次を Copilot App に貼り付ける。

```text
商品カードから商品詳細ページ（/product/:id）へ遷移する機能を実装したい。
詳細ページへ遷移後は一覧へ戻れるようにしたい。
```

「ルーティングはどの方式で実装する計画にしますか？」という問いかけには「react-router-dom の追加」の回答を選択する。

他にも「存在しない `/product/:id` にアクセスした場合は、どの挙動にしますか？」や「一覧へ戻る際は検索ワードの状態を維持しますか？」など、追加要件を確認することがあるが、回答は自由に選択する。※追加機能は少ない方が再現性が高い。

> [!TIP]
>**Plan モードの活用**
>
> Plan モードではあいまいな要件を確認する動作になりやすい。
> そのため、依頼主が認識していなかった要件や方針のすり合わせができ、齟齬が発生しづらい状態で実装に移ることが出来る。

後の手順で新しい依存の追加動作を確認するため、ルーティング方式に関する質問がなかった場合は Plan Summary のフェーズで以下のように意図的に提案をする。

```text
ルーティングは react-router-dom を使用する。
```

> [!NOTE]
> チーム運用規約の GitHub Flow に則り、計画を開始すると、自動で作業ブランチが作業内容に沿った名前（例: `feature/product-detail-routing`）へリネームされる。

### 2. プランを確認して実装する

View full Plan から計画と方針を確認します。プランが次の内容を満たしているか確認する。

- 商品詳細ルートと一覧へ戻る導線が含まれている。
- 追加で回答した要件が含まれている。
- 既存デザインに合わせたレイアウト整理が含まれている。
- 新しい依存を追加した背景に関する ADR 記録が含まれている。
- package-lock.json の同期が含まれている。
- 品質ゲート (npm test、npm build、画面遷移テスト) の実行が含まれている。

内容に合意したら、プランを承認し **Autopilot** で実装を進める。

> [!TIP]
>**Plan モード → Autopilot モードの活用**
>
> Autopilot は途中で確認を挟まず完了したら通知するフロー。
> 計画した todo タスクが完全に完了するまで自立的に動作する。
> その間、ユーザーは別のタスクに集中することができる。

#### 実装中の Agent の動作を確認する

実装中は、Copilot App が必要に応じて次の処理を行う様子を確認する。

- `git-workflow` スキル、`add-dependency-safety` スキルが読み込まれる。
- `react-router-dom` が Node 20 以上 / React 18 と互換であるか確認する。
- `npm install` で `package.json` と `package-lock.json` を同期する。
- 依存ファイルの編集後に Hook で不整合や環境互換性を確認する。
- 依存を採用した理由を `docs/decisions/` の ADR へ記録する。
- アプリ本体の実装だけでなくテストコードも実装する。
- `npm test` と `npm run build` を実行する。
- Browser Canvas で画面遷移を確認する。

#### 実装結果を確認する

- 商品カードから `/product/:id` へ遷移できる。
- 商品詳細ページから一覧へ戻れる。
- `react-router-dom` が `package.json` と `package-lock.json` に同期されている。
- `docs/decisions/` に依存判断の ADR が残っている。
- `npm test` と `npm run build` が成功している。
- 作業完了後、日本語のコミットメッセージ案と差分を提示している。

### 3. 変更差分とコミットメッセージ案を確認し、コミットする

コミットメッセージは証跡になるため、`git-workflow` スキルに HITL を仕込んでいる。

そのため、Interactive モードであればコミットメッセージ案と以下の選択肢が表示されるが、今回は Autopilot モードなので選択はスキップされ、メッセージ案の表示とコミット確認を促される動作となる。

- `このメッセージでコミット`
- `メッセージを修正`
- `コミットしない`

**changes** タブから変更内容を確認し、コミットメッセージ含め問題なければ、コミットを指示する。

> [!Note]
> ここでは以下を確認する。
> - 人が承認するまでコミットされない。
> - 承認した日本語メッセージでコミットが作成される。
> - コミット結果のみを報告して終了し、push や Pull Request 作成は行わない。

### 4. 互換性のないパッケージのインストール

このプロジェクトには、互換性のないパッケージをのインストールを防ぐガード (`dependency-guard` Hook) が用意されている。
この手順では意図的に互換性のないパッケージの依存追加を行い、Hook が作用するか確認する。

※ 手順 3 で PR Draft の作成を促されても先にこのステップを進める。

```text
異常系の動作検証です。
これは本番機能の依存追加ではなく、ガードレールを検証するための一時的なテストです。
次の手順だけを実施してください。

1. 検証用のローカル npm パッケージを tar ファイルで作成する。
 - package name: hook-test-node18-only
 - version: 1.0.0
 - engines.node: 18
 - 実装コードは不要
2. そのローカルパッケージを npm install で dependencies に追加する。
3. Node.js 20 以上と互換性がないことを意図したテストなので、互換性を理由にインストール前で中止せず、Hook の出力を待つ。
4. Hook から警告が追加コンテキストとして返ることを確認する。

この変更はコミット対象にしないため、確認が完了したらこの手順で作成・編集された箇所を元に戻してください。
```

![dependency-guard Hook が警告を返す画面](./images/01-dependecy-guard-hook-output.png)

> [!TIP]
> **Hook の活用**
> 
> Hook は特定のイベントが発生すると必ず発火し、用意したスクリプトを実行する。決定論的に実行結果をエージェントに注入できるためセンサーとしての役割を担う。

## 本ラボで期待する結果

- 作業内容に沿った名前へ、自動でブランチ名を変更する。
- 互換性チェックを済ませたうえで追加パッケージの依存追加を開始する。
- `react-router-dom` を含む同期済みの `package.json` / `package-lock.json` となっている。
- テスト、ビルドの動作を確認をした上で、実装完了と判断している。
- `docs/decisions/` に追加パッケージの選定背景が ADR として残っている。
- Hook が互換性のないパッケージインストールを検知する。
- 人の承認を得たうえでコミットする。

> 実装が終わらない場合は
> [講師ガイド](./instructor-guide.md#lab-01-復旧) の recovery asset を使う。

## Optional: 裏側の仕組みを確認する

Lab 02 の完了後、本ラボでガードレールがどのように動作したか、またどういうメリットがあるか確認する。

```text
このセッションについて、実際に残ったファイルや実行結果と対応付けて以下の説明とメリットを教えてください。
1. カスタムインストラクションと AGENTS.md が伝えたルール。
2. add-dependency-safely と git-workflow Skills が実行した手順。
3. Hook が検証した内容。
```
