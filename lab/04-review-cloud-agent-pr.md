# Lab 04: Cloud Agent の PR をレビューする

**テーマ:** Agent の完了報告ではなく、Issue・差分・テスト・CI の証跡で判断する

## シナリオ

Cloud Agent が検索バグの修正 PR を作成した。マージする前に、
Issue の受け入れ条件と PR の変更・テスト・CI を対応付けて人がレビューする。
CI が成功していても、要件どおりで使いやすいかを判断するのは人の役割。

## 前提条件

- Lab 03 の Issue に対して Cloud Agent が PR を作成していること。
  （未完了・利用不可の場合は [講師ガイド](./instructor-guide.md#lab-04-fallback) の共有 PR を使う）

## 手順

### 1. Issue と PR を対応付ける

```text
Issue #<番号> と、それに対して作成された Pull Request を確認してください。
受け入れ条件ごとに、PR のどの変更とどのテストが対応しているかを表にまとめてください。
CI（npm ci / npm test / npm run build）の状態と、依存関係に不要な変更が入っていないかも確認してください。
この時点では承認やマージはしないでください。
```

**確認する観点:**
- 半角・全角スペース区切りの複数キーワードが AND で扱われる。
- 連続・前後スペースを無視する。
- 空検索・単一キーワードの既存動作が維持されている。
- `src/lib/search.test.ts` に再現ケースと回帰テストがある。
- 依存追加や lockfile の不整合がない（ガードレールが Cloud Agent にも効いている）。

### 2. 動作を確認する（任意）

必要なら PR を別セッションで開き、Browser Canvas で `キャンプ テント` がヒットすることを確認する。

### 3. レビュー結果を返す

問題があれば、再現手順・実結果・期待結果・該当する受け入れ条件を含むコメントを投稿する。

```text
検証で見つかった次の問題について、再現手順・実際の結果・期待する結果・関連する受け入れ条件を含む
レビューコメントを作成して PR に投稿してください。解決方法は細かく指定しすぎないでください。

問題:
<ここに書く>
```

問題がなければ、確認内容を短くまとめて承認する。

```text
受け入れ条件、CI、テスト、動作確認をすべて通過しました。確認内容をまとめて PR を承認してください。
マージは行わないでください。
```

## 期待する結果 / 残る成果物

- 受け入れ条件と PR 変更・テストを対応付けたレビュー。
- 承認コメント、または再現可能な修正依頼コメント。

## まとめ

このワークショップで体験した SDLC:

```text
調査 → ローカル実装（依存追加でガードレール発火）→ Feature PR（CI で再確認）
→ 運用バグ Issue → Cloud Agent → Fix PR → 人によるレビュー
```

- **プロジェクトの前提（Custom Instructions）** が技術判断をそろえ、
- **SDLC ルール（AGENTS）** が Human-in-the-loop と Skill の起動条件を示し、
- **実行手順（Skills）** が依存追加と Git workflow を導き、
- **即時検証（Hook）** が事故を編集直後に検知し、
- **Pull Request 検証（CI）** がサーバー側で再確認し、
- **Issue と PR** が運用の証跡として残る。

会話に頼らず、リポジトリ自体がコンテキストとガードレールを持つ状態を「AI Ready」と呼ぶ。
